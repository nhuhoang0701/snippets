# Boost.Asio `io_context` Event Loop Methods — Complete Guide

## The Core Concept

All these methods are ways to **drive the event processing loop** of an `io_context`. They differ in three dimensions:
1. **Blocking behavior** — does it wait for work, or return immediately?
2. **How many handlers** — one, many, or all?
3. **Time limits** — indefinite, duration-based, or absolute-time-based?

---

## 1. `poll()` — Non-blocking, All Ready Handlers

**Signature:**
```cpp
std::size_t poll();
```

**Behavior:**
- Executes **all handlers that are currently ready** to run
- **Never blocks** — returns immediately if nothing is ready
- Returns the number of handlers executed

**Use Case:** Integrating with an external event loop (e.g., a game loop, GUI framework) where you want to process I/O events without blocking the main loop.

```cpp
while (game_running) {
    // Process all ready I/O handlers without blocking
    io_context.poll();
    
    // Continue with game logic
    update_game_state();
    render_frame();
}
```

**Important:** If `poll()` is called when no handlers are ready, it returns `0`. The `io_context` is **not stopped** — it just has nothing to do right now.

---

## 2. `poll_one()` — Non-blocking, At Most One Handler

**Signature:**
```cpp
std::size_t poll_one();
```

**Behavior:**
- Executes **at most one handler** that is ready to run
- **Never blocks** — returns immediately if nothing is ready
- Returns the number of handlers executed (0 or 1)

**Use Case:** Fine-grained control when you want to process exactly one event per iteration, or when integrating with systems that expect single-event processing.

```cpp
while (true) {
    std::size_t executed = io_context.poll_one();
    if (executed == 0) {
        // No handlers ready — do other work or sleep
        std::this_thread::sleep_for(1ms);
    }
    // Continue with other processing
}
```

---

## 3. `run()` — Blocking, All Handlers Until Stopped

**Signature:**
```cpp
std::size_t run();
```

**Behavior:**
- Blocks until **all work is finished** and there are no more handlers to dispatch
- Or until the `io_context` is explicitly stopped via `stop()`
- Returns the total number of handlers executed

**Use Case:** The standard way to run an async application. Typically called from one or more threads.

```cpp
// Standard usage
io_context.run(); // Blocks until all async work is done

// Multi-threaded usage
std::thread t1([&]{ io_context.run(); });
std::thread t2([&]{ io_context.run(); });
t1.join();
t2.join();
```

**Important:** If there's no work when `run()` is called, it returns immediately. Use `io_context::work` or `executor_work_guard` to keep it running.

---

## 4. `run_one()` — Blocking, At Most One Handler

**Signature:**
```cpp
std::size_t run_one();
```

**Behavior:**
- Blocks until **one handler has been dispatched**, or until the `io_context` is stopped
- Returns the number of handlers executed (0 or 1)

**Use Case:** When you need to process exactly one event and then do something else, but you're willing to wait for that event.

```cpp
while (!done) {
    io_context.run_one(); // Block until one handler completes
    
    // Check application state
    if (should_stop()) {
        io_context.stop();
    }
}
```

---

## 5. `run_for()` — Blocking, Time-Limited, All Handlers

**Signature:**
```cpp
template<typename Rep, typename Period>
std::size_t run_for(const std::chrono::duration<Rep, Period>& rel_time);
```

**Behavior:**
- Runs the event loop for **at most** the specified duration
- Executes as many handlers as are ready during that time
- Returns the number of handlers executed

**Use Case:** When you want to give the event loop a chance to process work, but only for a limited time (e.g., periodic processing in a game loop).

```cpp
while (game_running) {
    // Process I/O for up to 16ms (roughly one frame at 60 FPS)
    io_context.run_for(std::chrono::milliseconds(16));
    
    // Continue with game logic
    update_game_state();
    render_frame();
}
```

---

## 6. `run_until()` — Blocking, Absolute Time-Limited, All Handlers

**Signature:**
```cpp
template<typename Clock, typename Duration>
std::size_t run_until(const std::chrono::time_point<Clock, Duration>& abs_time);
```

**Behavior:**
- Runs the event loop until the specified **absolute time point**
- Executes as many handlers as are ready during that time
- Returns the number of handlers executed

**Use Case:** When you need to process events until a specific deadline (e.g., "process I/O until midnight").

```cpp
auto deadline = std::chrono::steady_clock::now() + std::chrono::seconds(5);

// Process I/O until the deadline
std::size_t handlers = io_context.run_until(deadline);

std::cout << "Executed " << handlers << " handlers\n";
```

---

## 7. `run_one_for()` — Blocking, Time-Limited, At Most One Handler

**Signature:**
```cpp
template<typename Rep, typename Period>
std::size_t run_one_for(const std::chrono::duration<Rep, Period>& rel_time);
```

**Behavior:**
- Blocks for **at most** the specified duration
- Executes **at most one handler** during that time
- Returns the number of handlers executed (0 or 1)

**Use Case:** When you want to wait for a single event, but with a timeout.

```cpp
// Wait for at most one handler, but only for up to 1 second
std::size_t executed = io_context.run_one_for(std::chrono::seconds(1));

if (executed == 0) {
    std::cout << "Timeout — no handler completed in time\n";
}
```

---

## 8. `run_one_until()` — Blocking, Absolute Time-Limited, At Most One Handler

**Signature:**
```cpp
template<typename Clock, typename Duration>
std::size_t run_one_until(const std::chrono::time_point<Clock, Duration>& abs_time);
```

**Behavior:**
- Blocks until the specified **absolute time point**
- Executes **at most one handler** during that time
- Returns the number of handlers executed (0 or 1)

**Use Case:** When you need to wait for a single event until a specific deadline.

```cpp
auto deadline = std::chrono::steady_clock::now() + std::chrono::seconds(5);

// Wait for at most one handler until the deadline
std::size_t executed = io_context.run_one_until(deadline);

if (executed == 0) {
    std::cout << "Deadline reached — no handler completed\n";
}
```

## Summary Comparison Table

| Method | Blocking? | Handlers Executed | Time Limit | Returns |
|--------|-----------|-------------------|------------|---------|
| `poll()` | ❌ No | All ready | None | Count |
| `poll_one()` | ❌ No | At most 1 | None | 0 or 1 |
| `run()` | ✅ Yes | All until stopped | None | Count |
| `run_one()` | ✅ Yes | At most 1 | None | 0 or 1 |
| `run_for()` | ✅ Yes | All during duration | Relative duration | Count |
| `run_until()` | ✅ Yes | All until time point | Absolute time | Count |
| `run_one_for()` | ✅ Yes | At most 1 during duration | Relative duration | 0 or 1 |
| `run_one_until()` | ✅ Yes | At most 1 until time point | Absolute time | 0 or 1 |

## Key Considerations

### Thread Safety
- Calling any of these methods from a thread that's already running one of them on the same `io_context` may cause **deadlock**.
- Multiple threads can safely call `run()` on the same `io_context` (thread pool pattern).

### Stopping the Loop
- `run()` and `run_one()` return when the `io_context` is stopped via `stop()` or when all work is done.
- `poll()` and `poll_one()` never stop the `io_context` — they just return if nothing is ready.
- After `stop()`, you must call `restart()` before calling `run()` again.

### Work Guards
If you need to keep `run()` alive even when there's no pending work, use `executor_work_guard`:

```cpp
auto work_guard = boost::asio::make_work_guard(io_context);
// io_context.run() will now block even with no pending handlers
// ...
work_guard.reset(); // Allow run() to return when work is done
```

---

## Practical Recommendations

| Scenario | Recommended Method |
|----------|-------------------|
| Standard async application | `run()` |
| Thread pool | Multiple threads calling `run()` |
| Integration with external event loop | `poll()` or `poll_one()` |
| Game loop / periodic processing | `run_for()` with frame duration |
| Waiting for a single event with timeout | `run_one_for()` |
| Processing until a specific deadline | `run_until()` or `run_one_until()` |
