# Multiple `io_context` Objects vs. Single `io_context` with Thread Pool

Both patterns are valid, but they serve different use cases. The choice depends on your workload characteristics and performance requirements.

---

## Pattern 1: Single `io_context`, Multiple Threads

This is the **default and most common** approach.

```cpp
boost::asio::io_context ioc;

// Thread pool of 4 threads, all calling run() on the same io_context
std::vector<std::thread> threads;
for (int i = 0; i < 4; ++i) {
    threads.emplace_back([&ioc] { ioc.run(); });
}

// All async operations are submitted to the same io_context
socket.async_read_some(buffer, handler);
timer.async_wait(handler);

// Handlers can execute on ANY of the 4 threads
```

### Advantages

| Benefit | Explanation |
|---------|-------------|
| **Automatic load balancing** | Boost.Asio's scheduler distributes work across threads automatically |
| **Simpler code** | One `io_context` to manage, no routing logic needed |
| **Work stealing** | If one thread is idle, it can pick up work from another thread's queue |
| **Fewer resources** | One event loop, one set of internal data structures |
| **Better for I/O-bound workloads** | Most async servers (web servers, chat servers, etc.) |

### Disadvantages

| Drawback | Explanation |
|----------|-------------|
| **Lock contention** | All threads access the same `io_context` internal queues |
| **Cache line bouncing** | Shared data structures cause cache invalidation across CPU cores |
| **Less predictable latency** | A handler might run on any thread, making timing less deterministic |

### When to Use

- **Most applications** — web servers, API backends, chat servers, game servers
- **I/O-bound workloads** where handlers are short and non-blocking
- **When simplicity matters** — you don't want to manage load balancing manually

## Pattern 2: Multiple `io_context` Objects, One Per Thread

Each thread has its own dedicated `io_context`.

```cpp
const int num_threads = 4;
std::vector<boost::asio::io_context> io_contexts(num_threads);
std::vector<std::thread> threads;

// Each thread runs its own io_context
for (int i = 0; i < num_threads; ++i) {
    threads.emplace_back([&ioc = io_contexts[i]] { ioc.run(); });
}

// You must manually decide which io_context gets which work
// Example: round-robin distribution
int next_ioc = 0;
for (auto& client : new_connections) {
    io_contexts[next_ioc].post([client] {
        client->start();
    });
    next_ioc = (next_ioc + 1) % num_threads;
}
```

### Advantages

| Benefit | Explanation |
|---------|-------------|
| **No lock contention** | Each `io_context` is accessed by only one thread |
| **Better cache locality** | Each thread's data stays in its own CPU cache |
| **Predictable latency** | Handlers always run on the same thread |
| **Better for CPU-bound work** | If handlers do significant computation, isolation prevents interference |
| **Fine-grained control** | You can prioritize certain `io_context` objects or pin workloads to specific cores |

### Disadvantages

| Drawback | Explanation |
|----------|-------------|
| **Manual load balancing** | You must decide how to distribute work across `io_context` objects |
| **No work stealing** | If one `io_context` is overloaded and another is idle, work isn't automatically redistributed |
| **More complex code** | Need routing logic, potentially dynamic load balancing |
| **Resource overhead** | Multiple event loops, multiple internal data structures |
| **Risk of imbalance** | Poor distribution strategy can leave some threads idle while others are overloaded |

### When to Use

- **CPU-intensive handlers** — if your completion handlers do heavy computation
- **Ultra-low latency requirements** — when you need predictable, minimal handler dispatch time
- **High-throughput systems** — when lock contention becomes a measurable bottleneck
- **Mixed workloads** — when you want to isolate different types of work (e.g., I/O vs. computation)
- **After profiling** — only switch to this pattern if profiling shows contention is a problem

## Pattern 3: Hybrid — `io_context` with `strand` for Serialization

A middle ground: single `io_context`, but use `strand` to ensure certain operations run on the same thread.

```cpp
boost::asio::io_context ioc;
boost::asio::io_context::strand strand1(ioc);
boost::asio::io_context::strand strand2(ioc);

// Handlers posted to the same strand are guaranteed to run sequentially
// (but not necessarily on the same thread)
strand1.post(handler1);
strand1.post(handler2);  // Runs after handler1 completes

strand2.post(handler3);  // Can run concurrently with strand1 handlers
```

This gives you serialization without the overhead of multiple `io_context` objects.

## Performance Comparison

| Metric | Single `io_context` | Multiple `io_context` |
|--------|---------------------|----------------------|
| **Lock contention** | Higher (all threads compete) | None (each thread has its own) |
| **Cache locality** | Lower (shared data structures) | Higher (per-thread data) |
| **Load balancing** | Automatic | Manual (your responsibility) |
| **Code complexity** | Low | High |
| **Throughput (I/O-bound)** | Excellent | Slightly better (if well-balanced) |
| **Throughput (CPU-bound)** | Good | Better (isolation prevents interference) |
| **Latency predictability** | Variable | More consistent |

## Real-World Guidance

### Start with Single `io_context`

For 90% of applications, a single `io_context` with a thread pool is the right choice:

```cpp
boost::asio::io_context ioc;
auto work_guard = boost::asio::make_work_guard(ioc);

// Thread pool size = number of CPU cores (or slightly more)
const int num_threads = std::thread::hardware_concurrency();
std::vector<std::thread> threads;
for (int i = 0; i < num_threads; ++i) {
    threads.emplace_back([&ioc] { ioc.run(); });
}

// Submit all async work to the same io_context
// Boost.Asio handles load balancing automatically
```

### Switch to Multiple `io_context` Only If

1. **Profiling shows lock contention** — use tools like `perf`, VTune, or Asio's built-in stats
2. **You have CPU-bound handlers** — handlers that do significant computation
3. **You need ultra-low latency** — and you're willing to manage load balancing
4. **You've measured the improvement** — don't optimize prematurely

### Example: When Multiple `io_context` Makes Sense

```cpp
// High-frequency trading system with mixed workloads
boost::asio::io_context market_data_ioc;      // Ultra-low latency market data
boost::asio::io_context order_execution_ioc;  // Order processing
boost::asio::io_context logging_ioc;          // Background logging

// Each has different latency/throughput requirements
// market_data_ioc might run on a dedicated core with real-time priority
```

## Summary

| Question | Answer |
|----------|--------|
| **Which should I use by default?** | Single `io_context` with multiple threads |
| **When should I use multiple `io_context`?** | When profiling shows contention, or for CPU-bound/mixed workloads |
| **Is one always better?** | No — it depends on your workload |
| **What's the risk of multiple `io_context`?** | Poor load balancing can make performance worse |
| **What's the risk of single `io_context`?** | Lock contention under extreme load (rare in practice) |

**Bottom line:** Start simple with a single `io_context`. Only move to multiple `io_context` objects if you've measured a real problem and understand the trade-offs. Most applications never need to make this switch.
