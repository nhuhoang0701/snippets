# The Proactor Pattern in Boost.Asio — Explained

## First: What Is the Proactor Pattern?

The name is a portmanteau: **Pro**active + **Actor**. It's an **asynchronous I/O design pattern** where:

- You **initiate** an operation and immediately get control back.
- The system **proactively notifies you** when the operation is done.
- You never block waiting for I/O — you get called back with the result.

This contrasts with the **Reactor pattern** (used by `select`/`poll`/`epoll` readiness notification), where the system tells you "this socket is **ready** to read/write" and you still have to do the actual I/O yourself. In the **Proactor** pattern, the system does the I/O for you and tells you "this operation is **complete**, here are the results."

> **Note:** On Linux, Boost.Asio actually uses the **Reactor pattern** internally (via `epoll`) and simulates Proactor behavior on top of it. On Windows, it uses true Proactor via **IOCP** (I/O Completion Ports).

---

## Mapping Each Element to Boost.Asio

Let me go through each element, explain what it means, and show what it maps to in real code.

---

### 1. Initiator

> "An I/O object that initiates the asynchronous operation."

**What it is:** The object you call the async method on. It's the "starter" — it kicks off the whole process.

**In Boost.Asio:** Any I/O object — `socket`, `steady_timer`, `serial_port`, `ssl::stream`, etc.

```cpp
boost::asio::steady_timer timer(io_context, 5s);
//     ^^^^^^^^^^^^^^^^^^^^^^^^
//     This is the Initiator
```

When you call `timer.async_wait(handler)`, the timer (initiator) forwards the request to the `io_context`.

---

### 2. Asynchronous Operation

> "A task to run asynchronously by the OS."

**What it is:** The actual I/O work that needs to happen — reading from a socket, waiting for a timer, writing data, etc. This is the **work itself**, not the object or the handler.

**In Boost.Asio:** The specific operation you requested:

| You Call | The Asynchronous Operation Is |
|----------|------------------------------|
| `socket.async_read_some()` | Wait for data to arrive on the socket and read it into a buffer |
| `socket.async_write()` | Wait for the socket to be writable and send data |
| `timer.async_wait()` | Wait until the timer's expiry time is reached |
| `acceptor.async_accept()` | Wait for an incoming connection |

The key point: this operation is handed off to the **OS** to perform. Your program doesn't sit in a loop checking "is data ready yet?" — the OS handles that.

---

### 3. Asynchronous Operation Processor

> "This executes the asynchronous operation and queues results in the completion event queue."

**What it is:** The component that actually **does the work** and, when done, puts the result into a queue. In practice, this is the **OS kernel** (or a thread pool managed by Asio for emulated async operations).

**In Boost.Asio:**

| Platform | Processor |
|----------|-----------|
| **Windows** | IOCP (I/O Completion Ports) — the kernel does the I/O and posts a completion packet |
| **Linux** | `epoll` + Asio's internal thread pool — `epoll` signals readiness, then Asio's reactor threads perform the actual I/O and enqueue the result |
| **macOS/BSD** | `kqueue` — similar to `epoll` |

The processor's job is:
1. Execute the I/O operation (or wait for the OS to complete it)
2. When done, create a "completion event" containing the result (success/error + data)
3. Push that event into the **completion event queue**

---

### 4. Completion Event Queue

> "An event queue where the asynchronous operation processor pushes events, and the asynchronous event [demultiplexer] dequeues them."

**What it is:** A **buffer/queue** that sits between the processor and the event loop. Completed operations are placed here as events, waiting to be picked up.

**In Boost.Asio:** This is an **internal queue inside `io_context`**. You never interact with it directly.

```
Processor finishes read on socket A
        │
        ▼
┌──────────────────────────┐
│  Completion Event Queue  │
│                          │
│  [event: socket A done]  │  ← pushed by processor
│  [event: timer B done]   │
│  [event: socket C done]  │
└──────────────────────────┘
        │
        ▼
  Picked up by the event demultiplexer / Proactor loop
```

This queue decouples the **producer** (OS/processor completing I/O) from the **consumer** (the event loop picking up results). Multiple completions can pile up in the queue without blocking anything.

---

### 5. Asynchronous Event Demultiplexer

> "This blocks the I/O context, waiting for events, and returning completed events to the caller."

**What it is:** The mechanism that **blocks** until one or more completion events are available in the queue, then **returns** them. "Demultiplexer" because it takes the single stream of events and routes them to the right place.

**In Boost.Asio:** This maps to the OS-level blocking wait:

| Platform | Demultiplexer Call |
|----------|-------------------|
| **Windows** | `GetQueuedCompletionStatus()` — blocks until IOCP has a completion |
| **Linux** | `epoll_wait()` — blocks until file descriptors are ready |
| **macOS/BSD** | `kevent()` — blocks until events are available |

When you call `io_context.run()`, internally it calls this demultiplexer in a loop:

```cpp
// Pseudocode of what io_context.run() does internally:
while (!stopped) {
    events = demultiplexer.wait();  // epoll_wait() / GetQueuedCompletionStatus()
    for (auto& event : events) {
        dispatch(event);  // call the right completion handler
    }
}
```

The demultiplexer **blocks the thread** until at least one event is ready — this is why `run()` blocks.

---

### 6. Completion Handler

> "A callable object that will process the results of the asynchronous operation."

**What it is:** The **callback** you provide — the lambda, function, or functor that gets invoked when the operation completes. This is your code that processes the result.

**In Boost.Asio:**

```cpp
timer.async_wait([](const boost::system::error_code& ec) {
    // This lambda is the Completion Handler
    if (!ec) {
        std::cout << "Timer expired!\n";
    }
});
```

The handler receives the result of the operation (typically an `error_code` and possibly data). It runs in the context of the thread calling `io_context.run()`.

---

### 7. Proactor

> "This calls the asynchronous event demultiplexer to dequeue events and dispatch them to the completion handler. This is what the I/O execution context does."

**What it is:** The **orchestrator** — the entity that ties everything together. It:
1. Calls the demultiplexer to get completed events
2. For each event, finds the corresponding completion handler
3. Dispatches (calls) that handler

**In Boost.Asio:** The `io_context` **is** the Proactor. When you call `io_context.run()`, you are running the Proactor loop.

```cpp
boost::asio::io_context ioc;
// ... set up async operations ...
ioc.run();  // ← This IS the Proactor in action
```

## The Complete Flow

Here's how all the pieces connect in a single async operation:

```
YOUR CODE                          BOOST.ASIO / OS
─────────                          ───────────────

1. timer.async_wait(handler)
   │
   ├─ Initiator (timer) ──────────► Registers operation with io_context
   │                                │
   │                                ├─ Asynchronous Operation: "wait 5 seconds"
   │                                │
   │                                ├─ Processor (OS/epoll) starts watching
   │                                │
   ▼                                │
   (your code continues)            │
                                    │
        ... time passes ...         │
                                    │
                                    ├─ Timer expires!
                                    ├─ Processor creates completion event
                                    ├─ Event pushed to Completion Event Queue
                                    │
                                    ▼
                            ┌────────────────────┐
                            │ Completion Queue   │
                            │ [timer done, ec=0] │
                            └────────────────────┘
                                    │
                                    │  io_context.run() is blocked here:
                                    ▼
                            ┌────────────────────┐
                            │ Demultiplexer      │
                            │ (epoll_wait /      │
                            │  GetQueued...)     │
                            │                    │
                            │ "Ah, an event!"    │
                            └────────────────────┘
                                    │
                                    │  Proactor (io_context) dispatches:
                                    ▼
                            ┌────────────────────┐
                            │ Completion Handler │
                            │ handler(ec)        │
                            │                    │
                            │ YOUR CODE RUNS     │
                            │ HERE               │
                            └────────────────────┘
```

## Summary Table

| Pattern Element | Boost.Asio Equivalent | Real-World Equivalent |
|-----------------|----------------------|----------------------|
| **Initiator** | `socket`, `timer`, `acceptor` | You picking up the phone and ordering food |
| **Async Operation** | The specific I/O task (read, write, wait) | The restaurant cooking your order |
| **Async Operation Processor** | OS kernel / IOCP / epoll + Asio threads | The kitchen staff |
| **Completion Event Queue** | Internal queue in `io_context` | The pickup counter where orders are placed |
| **Async Event Demultiplexer** | `epoll_wait()` / `GetQueuedCompletionStatus()` | The waiter checking "is any order ready?" |
| **Completion Handler** | Your lambda/functor | You receiving the food and eating it |
| **Proactor** | `io_context` (specifically `run()`) | The entire restaurant system coordinating everything |

## Why "Proactor" and Not "Reactor"?

| | **Reactor** | **Proactor** |
|---|---|---|
| **Notifies you when...** | I/O is **ready** (socket has data to read) | I/O is **complete** (data has been read into your buffer) |
| **Who does the I/O?** | **You** do it after notification | **The system** does it for you |
| **Pattern** | "Tell me when you're ready, I'll do the work" | "Do the work and tell me when it's done" |
| **OS support** | `select`, `poll`, `epoll`, `kqueue` | IOCP (Windows), `aio_read`/`aio_write` (POSIX AIO) |
| **Boost.Asio on...** | Linux (uses epoll, emulates Proactor) | Windows (uses IOCP, true Proactor) |

Boost.Asio presents a **uniform Proactor API** to you regardless of the underlying OS mechanism. That's the beauty of the abstraction — you write the same code on Linux and Windows, even though the internals differ.
