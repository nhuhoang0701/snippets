## Question 1: When a router sees an IP address, how does it know where to forward it?

Routers do not "know" where to forward a packet in the sense of having global knowledge of the entire Internet. Instead, they make **local, best-match decisions** using a **routing table** and the principle of **longest prefix match**.

Here is exactly what happens when a router receives a packet:

### 1. Extract the Destination IP
The router reads the destination IP address from the packet header. It does *not* look at the source address for forwarding decisions (source is only used for policies like NAT or filtering).

### 2. Longest Prefix Match Lookup
The router searches its routing table for the entry whose network prefix most specifically matches the destination IP. This is **not** first-match or exact-match — it is always the *longest* matching prefix. For example, given these routes:

-   `0.0.0.0/0` → Gateway A (default)
-   `192.168.0.0/16` → Interface eth1
-   `192.168.1.0/24` → Interface eth2

A packet destined for `192.168.1.50` matches all three, but `/24` is the longest prefix, so it wins and the packet goes out `eth2`.

### 3. Determine Next Hop vs. Direct Delivery
Once the matching route is found, the router checks whether the destination is on a directly connected network:
-   **Directly connected:** The destination IP is within a subnet attached to one of the router’s own interfaces. The router uses ARP (IPv4) or Neighbor Discovery (IPv6) to resolve the destination IP to a MAC address and delivers the frame directly.
-   **Not directly connected:** The route specifies a **next-hop gateway IP**. The router resolves *that* next-hop IP to a MAC address (via ARP/ND on the outgoing interface) and forwards the frame to the next hop. The router never needs to know the final destination’s location — only the next step.

### 4. Decrement TTL and Rewrite Layer 2
Before forwarding, the router:
-   Decrements the IP TTL by 1 (drops the packet and sends ICMP Time Exceeded if TTL reaches 0)
-   Recalculates the IP header checksum
-   Strips the incoming Layer 2 frame and builds a completely new Layer 2 frame with the appropriate source/destination MAC addresses for the outgoing link

### Where Does the Routing Table Come From?
The router doesn’t magically have this table. It is populated through:
-   **Directly connected networks:** Automatically added when an interface is configured with an IP/subnet.
-   **Static routes:** Manually configured by an administrator.
-   **Dynamic routing protocols:** OSPF, BGP, RIP, etc., which exchange reachability information with neighboring routers and algorithmically build the table. BGP, for example, propagates AS-path and prefix announcements across the entire Internet; OSPF floods link-state data within a domain.

### The Critical Insight
Routers operate on **hierarchical aggregation**, not flat lookup. The entire Internet routing system works because IP addresses are structured hierarchically (via CIDR), allowing routers to summarize millions of individual hosts into a small number of prefix entries. A core Internet router may have ~1 million BGP prefixes in its table, not billions of host entries. Without this hierarchy and longest-prefix-match semantics, scalable packet forwarding would be impossible.

In short: **A router forwards based solely on a local table lookup against the destination IP, choosing the most specific matching route, and sending the packet one hop closer to its destination — without ever needing to know the full path.**
