## Question 1: When a router sees an IP address, how does it know where to forward it?

Routers do not "know" where to forward a packet in the sense of having global knowledge of the entire Internet. Instead, they make **local, best-match decisions** using a **routing table** and the principle of **longest prefix match**.

Here is exactly what happens when a router receives a packet:

### 1. Extract the Destination IP
The router reads the destination IP address from the packet header. It does *not* look at the source address for forwarding decisions (source is only used for policies like NAT or filtering).

### 2. Longest Prefix Match Lookup
The router searches its routing table for the entry whose network prefix most specifically matches the destination IP. This is **not** first-match or exact-match — it is always the *longest* matching prefix. For example, given these routes:

-   `0.0.0.0/0` → Gateway A (default)
-   `192.168.0.0/16` → Interface eth1
-   `192.168.1.0/24` → Interface eth2

A packet destined for `192.168.1.50` matches all three, but `/24` is the longest prefix, so it wins and the packet goes out `eth2`.

### 3. Determine Next Hop vs. Direct Delivery
Once the matching route is found, the router checks whether the destination is on a directly connected network:
-   **Directly connected:** The destination IP is within a subnet attached to one of the router’s own interfaces. The router uses ARP (IPv4) or Neighbor Discovery (IPv6) to resolve the destination IP to a MAC address and delivers the frame directly.
-   **Not directly connected:** The route specifies a **next-hop gateway IP**. The router resolves *that* next-hop IP to a MAC address (via ARP/ND on the outgoing interface) and forwards the frame to the next hop. The router never needs to know the final destination’s location — only the next step.

### 4. Decrement TTL and Rewrite Layer 2
Before forwarding, the router:
-   Decrements the IP TTL by 1 (drops the packet and sends ICMP Time Exceeded if TTL reaches 0)
-   Recalculates the IP header checksum
-   Strips the incoming Layer 2 frame and builds a completely new Layer 2 frame with the appropriate source/destination MAC addresses for the outgoing link

### Where Does the Routing Table Come From?
The router doesn’t magically have this table. It is populated through:
-   **Directly connected networks:** Automatically added when an interface is configured with an IP/subnet.
-   **Static routes:** Manually configured by an administrator.
-   **Dynamic routing protocols:** OSPF, BGP, RIP, etc., which exchange reachability information with neighboring routers and algorithmically build the table. BGP, for example, propagates AS-path and prefix announcements across the entire Internet; OSPF floods link-state data within a domain.

### The Critical Insight
Routers operate on **hierarchical aggregation**, not flat lookup. The entire Internet routing system works because IP addresses are structured hierarchically (via CIDR), allowing routers to summarize millions of individual hosts into a small number of prefix entries. A core Internet router may have ~1 million BGP prefixes in its table, not billions of host entries. Without this hierarchy and longest-prefix-match semantics, scalable packet forwarding would be impossible.

In short: **A router forwards based solely on a local table lookup against the destination IP, choosing the most specific matching route, and sending the packet one hop closer to its destination — without ever needing to know the full path.**

## Question 5: Speculate on why IP is below TCP in the layered model. Why does the TCP header go on after the IP header and not the other way around?

The ordering of IP below TCP is not arbitrary; it is a direct consequence of the **separation of concerns** between *where* data goes and *how reliably* it gets there. The IP header must come first because routers need to make forwarding decisions based solely on Layer 3 information, without ever needing to parse or understand transport-layer semantics.

Here are the fundamental reasons for this ordering:

### 1. Routers Only Need IP
Routers operate at Layer 3. When a router receives a packet, it inspects the destination IP address, performs a longest-prefix-match lookup, decrements TTL, and forwards. It does **not** care about ports, sequence numbers, acknowledgments, or window sizes. If the TCP header came first, every router would have to parse past variable-length TCP options just to find the IP addresses — adding latency, complexity, and fragility to every hop across the Internet. By placing IP first, routers can extract what they need from fixed offsets in the first 20 bytes and ignore everything that follows.

### 2. IP Is Transport-Agnostic
IP carries many different transport protocols: TCP, UDP, ICMP, GRE, ESP, etc. The IP header’s `Protocol` field tells the receiving host which upper-layer handler should process the payload. This multiplexing only works if IP is the outer envelope. If TCP were outside, you couldn’t carry UDP or ICMP inside a TCP segment — the layering would collapse into a single-protocol tunnel rather than a general-purpose internetworking layer.

### 3. End-to-End vs. Hop-by-Hop
TCP is an **end-to-end** protocol. Intermediate routers never inspect, modify, or act on TCP headers (with rare exceptions like middleboxes, which are widely considered architectural violations). IP is a **hop-by-hop** protocol; every router along the path processes it. Placing the hop-by-hop header on the outside and the end-to-end header on the inside reflects exactly who needs to see what. This is a direct application of the **end-to-end principle**: functionality that only endpoints require should not be exposed to the network core.

### 4. Fragmentation and Reassembly Semantics
When an IP packet is too large for a link’s MTU, routers fragment it at the IP layer. Each fragment retains the full IP header (with adjusted offset and MF flags) but only the *first* fragment contains the TCP header. If TCP were outside, fragmentation would break TCP entirely — intermediate fragments would lack port and sequence information, making reassembly impossible without reconstructing the entire transport state at every router. With IP on the outside, fragments are self-contained Layer 3 units that can be reassembled before TCP ever sees them.

### 5. Hardware and Pipeline Efficiency
Network interface cards and ASICs are designed to parse headers in wire order. Having a fixed, well-known IP header at the start allows hardware to begin routing lookups in parallel with DMA transfers, before the rest of the packet has even arrived. A TCP-first layout would force serial parsing and stall forwarding pipelines.

### The Core Insight
The layered model is not just conceptual abstraction — it is a **physical constraint on packet processing**. IP is below TCP because the network infrastructure *must* be able to forward packets using only IP-level information, while transport reliability remains entirely invisible to the forwarding plane. Reversing the order would couple routing to transport semantics, destroy protocol multiplexing, break fragmentation, and make scalable hardware forwarding impractical. The header order is the wire-format embodiment of the architectural boundary between the network and the endpoints.
